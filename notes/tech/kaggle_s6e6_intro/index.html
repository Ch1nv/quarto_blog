這篇在解什麼問題

望遠鏡掃過天空時，畫面上每一個亮點看起來都差不多：一個點、一團光。但這些點背後可能是三種完全不同尺度的東西。

  • STAR（恆星）：我們銀河系裡的一顆星，距離近、紅移幾乎是零。
  • GALAXY（星系）：一整個遙遠的星系，距離遠、紅移中等。
  • QSO（類星體 / quasar）：遙遠星系中心極度明亮的活躍核，外觀像一個點，所以名字才叫「quasi-stellar」——看起來像恆星，實際上在非常遠的地方，紅移通常很高。

麻煩的地方就在這裡：QSO 在影像上長得像恆星，物理上卻像遙遠星系。 光看一個亮點分不出來，得靠數字——它在天空中的位置、在不同濾鏡下的亮度、還有紅移——才能判斷它到底是哪一類。這正是 SDSS 這類巡天計畫每天在大規模做的事，也是建立天文星表的基礎工作。

這個 Kaggle 競賽（Playground Series S6E6）把這件事包成一個乾淨的監督式三分類問題：給你一張表，每一列是一個天體與它的觀測特徵，要你預測它是 GALAXY、QSO 還是 STAR。資料是從真實的恆星分類資料集合成出來的，所以欄位有真實的天文意義，但測試答案不會外流。

評分方式

提交格式是 id,class 的硬標籤（不是機率）：

id,class
123456,GALAXY
123457,STAR
123458,QSO

評分指標是 balanced accuracy，也就是「各類別 recall 的平均」：

\[ \text{Balanced Accuracy} = \frac{1}{K}\sum_{k=1}^{K} \text{recall}_k \]

ImportantBalanced Accuracy

一般 accuracy 會被最大的類別主導——只要把所有東西都猜成 GALAXY，光靠它佔了三分之二就能拿到不低的分數。balanced accuracy 不吃這套：它把每一類的 recall 等權平均，最小的 STAR（只佔 14%）和最大的 GALAXY 一樣重要。

換句話說，這個競賽從評分一開始就在逼你正視類別不平衡。後面 EDA 看到的分布，不只是「順便看一眼」，而是直接決定你怎麼設計模型。

（指標以競賽頁面的 Evaluation 分頁為準，建議下手前再確認一次。）

資料長什麼樣子

三個檔案的規模：

資料集 列數 欄數
train 577,347 12
test 247,435 11
sample_submission 247,435 2

train 比 test 多的那一欄就是答案 class。id 只是識別碼，不該當成建模特徵。

剩下十個欄位，八個數值、兩個類別，各自帶著天文意義：

欄位 型別 直覺
alpha, delta 數值 天球座標（赤經、赤緯），天體在天空中的位置
u, g, r, i, z 數值 五個濾鏡波段下的星等（亮度），SDSS 標準測光系統
redshift 數值 紅移，和天體的距離、退行速度與物理類型高度相關
spectral_type 類別 光譜類型
galaxy_population 類別 星系族群，例如 Red_Sequence 或 Blue_Cloud
class 目標 GALAXY / QSO / STAR

EDA：我從這份資料看到什麼

類別不平衡分布

類別 筆數 佔比
GALAXY 377,480 65.38%
QSO 117,143 20.29%
STAR 82,724 14.33%

GALAXY 一個類別就佔了約三分之二，STAR 最少、只有約 14%。接上前面的指標：因為評分是 balanced accuracy，模型在 STAR 上犯的錯，跟在 GALAXY 上犯的錯一樣會被放大。所以我評估模型時不會只看整體 accuracy，而會盯著 confusion matrix 和每一類的 precision / recall，確認它沒有偷懶把東西都丟給 GALAXY。

缺失值：乾淨

目前 starter 統計顯示所有建模欄位的缺失值都是 0。

這代表第一版 baseline 不必特別補值。不過 pipeline 裡我仍保留 SimpleImputer——之後一旦做了特徵工程、產生新的缺失值，整條流程才不會直接掛掉。這是一種「現在用不到、但留著很便宜」的保險。

數值特徵：座標、亮度與一個顏色的想法

天球座標的範圍大致鋪滿整個天區。alpha（赤經）幾乎覆蓋完整一圈，delta（赤緯）落在約 −18° 到 79° 之間：

統計量 alpha delta
min 0.0117 −17.9670
median 188.6815 21.4844
max 359.9998 79.1583

五個亮度波段的平均星等，從 u 到 z 是單調遞減的：

波段 平均星等
u 22.4419
g 21.0073
r 19.9628
i 19.3789
z 19.0411

單一波段的亮度其實受距離、儀器等因素影響很大，真正有區辨力的常常是波段之間的差——也就是天體的「顏色」。所以一個很自然的下一步，是把相鄰波段相減做出顏色特徵：

train["u_g"] = train["u"] - train["g"]
train["g_r"] = train["g"] - train["r"]
train["r_i"] = train["r"] - train["i"]
train["i_z"] = train["i"] - train["z"]

這類顏色差通常比單一星等更能捕捉天體的光譜形狀，在這種分類問題裡往往是很有力的特徵。

redshift：右偏，而且可能是 QSO 的關鍵

統計量 值
mean 0.7231
std 0.8101
min −0.0100
25% 0.1811
50% 0.4975
75% 0.8814
max 7.0108

平均值（0.72）明顯大於中位數（0.50），最大值（7.01）也遠超過 75% 分位數——這是一個右偏分布，有一條長長的高紅移尾巴。

把它接回物理：恆星紅移幾乎是零（min 出現 −0.01，是量測噪聲級別的小負值），星系紅移中等，而高紅移那條尾巴很可能就是 QSO。如果真是這樣，redshift 會是區分這三類最有力的單一特徵之一。這也是我接下來想做的第一個分析：把 redshift 在三類之間的分布畫出來比較。

Tip

高紅移、長尾的分布，常見的處理是 log1p 轉換或 clipping，讓模型不被極端值牽著走。這會是調分數時值得一試的方向。

類別特徵：兩個已經帶天文意義的欄位

兩個類別欄位本身就是天文分類的結果，預期對模型有幫助：

  • spectral_type 有 4 種，最多的是 M，共 303,323 筆（約 52.54%）。
  • galaxy_population 有 2 種，最多的是 Red_Sequence，共 319,565 筆（約 55.35%）。

要確認它們到底有沒有用，最直接的方式是看它們和 class 的交叉表——每一種類型底下，三類的比例長怎樣：

pd.crosstab(train["spectral_type"], train["class"], normalize="index")
pd.crosstab(train["galaxy_population"], train["class"], normalize="index")

如果某個 spectral_type 幾乎只對應某一類，那它就是很乾淨的訊號。

目前的 baseline 與接下來想試的

第一版 baseline 用的是 HistGradientBoostingClassifier，前處理很單純：數值欄位用中位數補值，類別欄位用眾數補值並做 OrdinalEncoder 編碼：

本地修改 notebook
  -> kaggle kernels push
  -> Kaggle 雲端執行
  -> kaggle kernels output 抓回 outputs/

有了能動的管線之後，接下來想做的事，大致照優先順序：

  1. 加入顏色差特徵（u-g, g-r, r-i, i-z）。
  2. 把 redshift 在 GALAXY / QSO / STAR 之間的分布畫出來，驗證「高紅移 ≈ QSO」的猜測。
  3. 建 confusion matrix，確認模型沒有偏向最大類別 GALAXY。

再往上提分數，候選清單還有：換 LightGBM / XGBoost / CatBoost；對 redshift 做 log 轉換或 clipping；針對不平衡調 class weight 或決策門檻；以及把兩個類別欄位改成 one-hot，或乾脆用 CatBoost 原生的 categorical 處理。
